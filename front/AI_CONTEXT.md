# AI Context — Front (Flutter)

> Front-specific context for AI agents. System-wide contract, wire format, and cross-component invariants: [`../AI_CONTEXT.md`](../AI_CONTEXT.md).

---

## Role

Offline-first Flutter client (Android, iOS, Web from one codebase). All authenticated traffic goes through the back's single endpoint `POST /v1/sync`: scope-keyed cursors + pending mutations out, authoritative `authorized_scopes` + one `ScopeSyncResult` per scope + per-mutation outcomes in.

> **Contract note (2026-06):** `Owner` and `Member` do not carry a `sub` field on the wire — front mappings use `owner_id` / `member_id` as entity identifiers. The JWT `sub` is auth-scoped metadata only (e.g. private feed keys `member:{sub}` / `owner:{sub}`).

---

## Layered architecture (`lib/`)

```
domain/        pure Dart, zero Flutter/IO — models, sync types, auth/server contracts
data/          adapters (drift, dio, repositories, auth, push) — depends on domain/ only
presentation/  BLoC + go_router + Material 3 screens — depends on domain/ only
```

`presentation` never imports `data` directly: repositories arrive via `RepositoryProvider`, blocs via `BlocProvider`.

- **`domain/model/`** — one Freezed model per synced entity, snake_case via `@JsonKey`. `organization_member_view.dart` holds pure selector functions (volunteer dashboard, history, coordinator displays).
- **`domain/sync/`** — `EntityType` enum + wire-name map, `EntityPayload` (sealed), `ClientMutation` (`tmpIdPrefix = 'tmp_'`), `MutationOutcome`/`MutationErrorCode`, `SyncRequest`/`SyncResponse`, `ScopeSyncResult`, scope-key helpers in `sync_scope.dart` (incl. `memberScopeKey(sub)` / `ownerScopeKey(sub)`).
- **`data/local/database.dart`** — drift schema: one table per synced entity, `sync_cursors` keyed by **scope key**, `pending_mutations` (with local-only `scope_key` for pruning/remap). Most entity tables store full JSON blobs.
- **`data/network/`** — `buildSyncDio` (auth interceptor, signOut on 401) / `buildPublicDio`; `SyncApi` (POST /v1/sync); `PublicApi` (public bootstrap endpoints + activate, typed conflict exceptions); `AdminApi` (only `searchProducers` — the one authenticated REST call).
- **`data/auth/`** — `GoTrueAuthService` (direct REST via dio) and `CognitoAuthService` (`amazon_cognito_identity_dart_2` behind a `CognitoSessionGateway`; `USER_PASSWORD_AUTH`, access token only — never the idToken) behind one `AuthService` interface; `AuthTokenStorage` (mobile: SharedPreferences; web: sessionStorage, localStorage only with "remember me"); `JwtClaims` (unsigned payload decoder — back is the trust boundary); `RememberedUserContextStorage` (email/server prefill, separate from the session).
- **`data/server/`** — `ServerCatalog` backed by static `server_presets.dart` (temporary bootstrap), `ServerConfigStorage` (persisted selected config), `ServerDiscoveryDocument` + `DiscoveryDocumentServerCatalog` (federation-ready).
- **`data/repositories/`** — one repository per entity, mirroring `ProductTypeRepository`: optimistic local write + `ClientMutation` enqueue (some flows are deliberately non-optimistic and wait for the sync outcome, e.g. join-request review, owner/producer lifecycle).
- **`data/sync/sync_repository.dart`** — the orchestrator (see below).
- **`data/push/`** — `PushRegistrationService` + `PushTokenSource` (`FcmPushTokenSource` wraps `firebase_messaging`, degrades gracefully without Firebase config).
- **`presentation/`** — screens follow the UI specs under `documentation/feature/fr/ui/` (the spec is the contract — see `front/AGENTS.md`). Routing in `presentation/router.dart` (go_router, auth-guarded redirects, `/login?from=...` intent preservation).
- **`presentation/delivery/`** — shared delivery-display building blocks (reuse instead of re-duplicating): `DeliveryCard` (one card for the member planning + dashboard, `DeliveryCardVariant.planning`/`dashboard`), `delivery_format.dart` (date/time formatters), `delivery_status_chip.dart` (`DeliveryStatusChip`/`SlotStatusChip`), `delivery_coordinators.dart` (`CompactCoordinatorsLine`/`CoordinatorsSection`), `delivery_registration_actions.dart` (`RegisterButton`/`UnregisterButton` + actions). The pure slot/registration selectors live in `domain/model/organization_member_view.dart`. **`deliveryVolunteerStaffing(delivery, {mainContractIds})` is the single source of truth for the "N/M bénévoles" counter** — sum of (`currentRegistrations`, `requiredVolunteers`) over all non-cancelled STANDARD **and** EARLY slots; every display site routes through it (`DeliveryCard`, `volunteer_dashboard_section`, `deliveryVolunteerSummary`, the coordinator tracking progress bar). **Only main contracts count** (`Contract.isMainContract`): callers pass `mainContractIds` (derived via `mainContractIdsOf(contracts)`) so secondary contracts (eggs, fruit…) contribute no volunteer need — fallback counts every contract when none of the delivery's links is main (legacy data). **Coordinators never count**: the delivery-wide coordinator union (`deliveryCoordinatorIds`, every linked `DeliveryContract.coordinators`) is excluded from `current` via `activeRegistrationsExcluding`, even when a coordinator of one contract registers on another contract's slot. `deliverySlotStatus` and the coordinator tracking "Bénévoles présents" count apply the same main-filter + coordinator-union rules. Volunteer slots are materialised on the **main contract link only** (`defaultVolunteerSlots` is attached to the main link by the delivery form and weekly generation; **no slot is created when no linked contract is main** — the delivery form also hides the volunteer minimum field + slot-time block in that case, lists only main-contract slots, and prunes legacy empty volunteer slots off non-main contracts on save). `registerToSlotAction` likewise targets a **main contract's** slot (`mainContractIds` passed from the cards via `RegisterButton`), so a volunteer never lands on a secondary contract; legacy fallback to every contract only when none is flagged main. Slot capacity reads `slot.requiredVolunteers` for both kinds (`slotCapacity`); the EARLY slot is materialised with `requiredVolunteers` = the resolved early capacity (delivery override, then `DeliveryTemplate.earlySlot.maxVolunteers`), so the slot row is the override-aware source and no template lookup is needed at read time. A `Delivery` also carries optional `standardEndTime` / `volunteerArrivalTime` / `earlySlot` overrides of the template's slot times (resolution: delivery override → template → hard-coded default), materialised by `defaultVolunteerSlots` in `domain/model/delivery_slots.dart`; the coordinator delivery form authors them (`_SlotTimesBlock`, "Horaires des créneaux").

## Sync machinery

- **`SyncRepository` applies the whole response in one drift transaction**: treats `authorized_scopes` as authoritative discovery (evicts disappeared scopes + their pending mutations), applies bootstrap as a full authoritative resync even after an incremental request, remaps `tmp_*` ids, rewrites still-pending mutations referencing them, drains acknowledged mutations. REJECTED outcomes surface via `SyncSuccess.rejectedMutations`.
- **Handler registry** — `data/sync/entity_sync_handler.dart`: one `EntitySyncHandler` per `EntityType` owning snapshot eviction, upsert, delete, and `tmp_*` remap. The registry asserts full coverage at construction — **adding an `EntityType` without a handler crashes at startup**.
- **`tmp_*` remap** — when `MutationOutcome.serverEntityId != tmpId`, the row's primary key is rewritten (delete + insert in-transaction). No-account producer remap also rewrites embedded `Organization.producers[]` references (not `Organization.products[]`). Contract remap (`ContractSyncHandler`) also rewrites `Organization.deliveries[].contracts[].contractId` in pending `OrganizationPayload` mutations AND in the cached `organizations` table (via `db.remapContractId`). `BasketExchange` nested requests: `serverEntityId` carries the aggregate root id — the allocated `requestId` is recovered by diffing `requests` after the next sync.
- **Sync triggers** — app start, manual refresh, post-mutation flush (`SyncEvent.mutationApplied()` dispatched by screens), `connectivity_plus` reconnection (none→online transitions only). No periodic polling.
- **Offline failures are classified, not generic** — `SyncRepository` maps transport-level `DioException`s (connection error/timeouts, `unknown` wrapping a SocketException) to `SyncOutcome.networkFailure()` → `SyncState.offline()`; HTTP error responses stay `SyncOutcome.failure`. UI: `SyncStatusBanner` shows a reassuring "Serveur injoignable…" banner (retry only, no report), and `SyncOfflineListener` (mounted above the `MaterialApp` in the authenticated shell, snackbar via `scaffoldMessengerKey`) surfaces the same message on any screen. Shared copy in `presentation/sync/sync_messages.dart`.
- **Tenant resolution** — `producerAccountId == sub` for every user (no org claim in tokens). Repositories resolve the `organizationId` lazily: exact match first, then the org id derived from the `organization:*` sync cursor — so non-producer users see empty data until the first sync completes.

## Key decisions

- **Freezed + json_serializable** with global `include_if_null: false` (matches back `explicitNulls = false`); snake_case via explicit `@JsonKey(name: ...)`. Wire compatibility pinned by `test/domain/sync_wire_format_test.dart`.
- **Hand-rolled sealed parents for `EntityPayload` / `MutationOp`** — Freezed 3.x drops the `type` discriminator for single-variant unions, so the parents are manual `sealed class`es with explicit `fromJson` switch / discriminator-injecting `toJson`; variants stay Freezed.
- **drift** for the local cache: type-safe SQL, first-class transactions (needed for remap + atomic snapshot apply), cross-platform via `drift_flutter`. Web assets shipped manually: `web/sqlite3.wasm` + `web/drift_worker.dart.js`. **Corruption self-heal (web)**: `database_open_web.dart` opens a throwaway probe connection (`enableMigrations: false`) and runs `PRAGMA quick_check` via `isDatabaseHealthy` (`database_integrity.dart`); on a malformed image (e.g. a Firefox multi-tab write race on the `unsafeIndexedDb` storage) it reports a groupable Sentry event (`'Local cache corrupt — recreated'`, warning, with `local_cache` context) for tracing, then `deleteDatabase`s the stored copies and recreates an empty cache — safe because the cache is server-authoritative and the next sync rebuilds it. Recovery happens on (re)load. The open/recover **control flow** is extracted into the platform-agnostic `openDatabaseWithRecovery` (`database_recovery.dart`, unit-tested) so the wasm/Sentry specifics stay in the web file.
- **Auth session semantics** — lazy single-flight refresh within 30s of expiry; refresh failure ⇒ `Unauthenticated` + storage cleared; back 401 ⇒ `signOut()`. `SyncBloc` is provided keyed by `producerAccountId`, so login/logout recreates it cleanly.
- **App boots on the public home** (`/`); authenticated users are redirected to `/product-types`. Server selection is a secondary screen reachable from login; switching server clears auth + config.
- **Push registration is server-authoritative** — `PushRegistrationService` registers `DeviceToken`s only on the private feeds present in persisted sync cursors (never guessed from claims), idempotent by token, bound to FCM token rotation, fired on every `SyncSucceeded`. `firebase_options.dart` ships placeholders (build/boot only — real delivery needs a real Firebase project). `unregisterCurrentDevice` exists but is not auto-fired on logout (deferred).
- **`BasketExchange` is a reciprocal swap** — `BasketExchangeRequest` carries `proposedDeliveryId` / `proposedContractId` (the requester's counter-basket, required at submit; `SubmitRequestDialog` picks it from the requester's eligible deliveries). The offerer **validates** one request (`acceptRequest`) or **refuses it individually** (`refuseRequest`, offer stays OPEN — now supported by the back). Pure selectors in `domain/model/basket_exchange_view.dart` (`basketExchangeSummaryFor`, `committedDeliveryIdsFor`, `pendingRequestCount`) feed the home `BasketExchangeDashboardCard` and the all-members overview at `/basket-exchange/overview` (CSV export via `saveDatabaseExportFile`). `committedDeliveryIdsFor` mirrors the back's `isBasketCommitted` (offered + both sides of a settled swap, given **or** received) and gates the deliveries offered by both `ProposeBasketExchangeDialog` (new offer) and `SubmitRequestDialog` (counter-delivery). Notification deep links are followed by `NotificationInboxScreen` on tap.
- **Shared baskets (panier partagé)** — `Contract.sharedBaskets: List<SharedBasket{sharedBasketId, memberIds, anchorDeliveryId?}>` overlay (members keep their individual `ContractMember`). Pure selectors in `domain/model/shared_basket_view.dart` (`sharedBasketPickerFor` mirrors the back alternation formula, `contractDeliveriesOrdered`, `pickupDeliveriesFor`, `sharedBasketForMember`, `coSharersFor`). The coordinator authors them on `coordinator_member_contracts_screen.dart` (a "PANIER PARTAGÉ" row + `_SharedBasketDialog` per assigned contract, writing the overlay via `ContractRepository.update` with a `tmp_` `sharedBasketId`). `ContractSyncHandler` needs **no remap** for the nested id — the authoritative `ContractPayload` overwrites the cached contract on the next sync. Members see the alternation read-only on `/contracts` (`_SharedBasketInfo`: "1 distribution sur N" + their pickup dates) and on the planning `DeliveryCard` (planning variant, fed `contracts:` from `member_delivery_plan_screen`; `_sharedBasketLines` shows "c'est votre tour" or "récupéré par {famille}" per linked shared contract). `MutationErrorCode.invalidSharedBasket` mirrors the back. **Exchange interaction**: `memberHoldsBasketOn` (in `shared_basket_view.dart`, mirrors back `Contract.holdsBasketOn`) gates the `BasketExchange` dialogs — `BasketExchangeBloc` now also watches `ContractRepository` (exposed as `state.contracts`), and `ProposeBasketExchangeDialog` / `SubmitRequestDialog` hide the weeks/contracts where the member shares the basket but it is not their turn (the back enforces the same via `FORBIDDEN`).
- **Notifications** are server-authoritative: `NotificationRepository` only `markRead` / `archive`; inbox at `/notifications`.
- **Admin alert-copy customisation** — `/preferences` shows an extra "Personnalisation des alertes" card to `UserRole.admin` only (router wires an `AlertTemplatesBloc` alongside `UserPreferencesBloc`; `UserPreferencesScreen(showAlertTemplates: true)`). It edits `Organization.notificationOverrides` (`Map<NotificationCategory, NotificationCopyOverride>`) per `kCustomisableAlertCategories` and saves via `OrganizationRepository.updateNotificationOverrides` (blank entries dropped). `Organization` round-trips the map automatically through `OrganizationPayload`.
- **Organization backup / migration** — `/preferences` shows an admin-only "Sauvegarde & migration" card (`UserPreferencesScreen(backupOrganizationId: <tenant>)`, wired by the router for `UserRole.admin`). Export (all platforms) calls `AdminApi.exportOrganization(orgId)` → saves the raw JSON via `saveDatabaseExportFile` (reused download/save helper). Import is **web-only in V1** (no file-picker dependency on mobile): the button is hidden unless `kIsWeb`, reads a file via `pickJsonFile()` (`data/local/json_file_picker*.dart`, conditional web/stub import), POSTs through `AdminApi.importOrganization` then nudges `SyncBloc` to pull the restored data. When the result carries `warnings` (e.g. members skipped because their email already belongs to an instance owner), they are shown in a dialog instead of the plain success snackbar. The archive stays **opaque** on the client (no Dart model) — it is a pass-through file. Back endpoints: `GET/POST /v1/admin/organizations/{id}/export|import` (see root `AI_CONTEXT.md`).
- **Bulk "ask to connect"** — `/members` shows a `_PendingConnectionBanner` when ≥1 `PENDING_ACTIVATION` invitation; its dialog optionally overrides the email subject/body and dispatches `UserManagementEvent.resendAllPendingRequested`, which calls `MemberInvitationRepository.resend(... customEmailSubject/Body)` per pending invite then one sync. `MemberInvitation` carries nullable `customEmailSubject` / `customEmailBody`.
- **Per-delivery basket composition** — informative-only basket contents (see root `AI_CONTEXT.md` invariant). Producer manages the source `ItemType` catalog (`ItemType{id, name, imageSvg?}`, **SVG-only** inline markup) at `/product-types/:id/items` (`ItemTypesScreen`/`ItemTypesBloc`; entry: a "Catalogue de composants" card on `ProductTypeFormScreen`, edit mode). Coordinator composes a delivery at `/coordinator/deliveries/:deliveryId/description` (`DeliveryDescriptionScreen`/`Bloc`; entry: a "Composition du panier" card on the time-slot form `_BasketCompositionBlock`, existing deliveries only — editor loads from the org cache). **`DeliveryItem` stays lean** (`itemTypeId` + optional `weight` + tiny `name` snapshot) — the **SVG is never duplicated per item**. On save the bloc **merges** the used components' `ItemType`s (by id) into the flat org-level catalog `Organization.itemTypes` via `OrganizationRepository.updateDeliveryDescription(..., itemTypes:)` in one org `Upsert`; members get the catalog through the `organization` scope. SVG is rendered everywhere via the shared `ItemTypeSvgIcon` (`SvgPicture.string`, `flutter_svg`). Members see it read-only on the planning `DeliveryCard` via `BasketCompositionSection` (collapsible, grouped by basket size; the SVG is resolved by `itemTypeId` from `org.itemTypes`, name + optional weight from the item).

## Adding a synced entity (recipe)

Mirror a new back `EntityType` in this order (see also the `sync-entity` skill):

1. `domain/sync/entity_type.dart` — enum value + wire-name map entry.
2. `domain/model/<entity>.dart` — Freezed model, snake_case `@JsonKey`s.
3. `domain/sync/entity_payload.dart` — payload variant + `fromJson` switch branch + discriminator `toJson`.
4. `data/local/database.dart` — drift table + CRUD + `clearXForTenant`, schema bump + migration.
5. `data/sync/entity_sync_handler.dart` — handler + registration in `buildEntitySyncHandlers()`.
6. `data/repositories/<entity>_repository.dart` — mirroring `ProductTypeRepository`.
7. Screens + routes.
8. Tests: wire round-trip, drift CRUD, sync paths, repository writes.
9. `dart run build_runner build --delete-conflicting-outputs`.

## Configuration & design

- No `--dart-define` keys. Bootstrap from the static in-binary `serverPresets` (the `id` is the persistence key — keep it stable); selected `ServerConfig` persisted in SharedPreferences.
- Theme: `ColorScheme.fromSeed(seedColor: Colors.green)` + Material 3 — only the three raw `Colors.*` accents on the home action cards deviate. Brand assets: `assets/logo.svg`, `assets/wordmark.svg` (`flutter_svg`). Invoke the `amap-en-ligne-design` skill before any UI work.

## Tests

Layout under `test/` mirrors `lib/` (`domain/`, `data/`, `presentation/`, plus `acceptance/` for scripted offline-first stories with real drift + scripted `SyncApi`). Conventions and pitfalls: `front/AGENTS.md` (Testing Rules). Goldens not yet introduced.
